Día 4: Comunicación de proyectos de ML

Quarto como sistema unificado de documentación

Workshop IEEE 2026 — Guía del instructor

1) Orden del día

Día 4: Comunicación de proyectos de ML.

Objetivo de la sesión:

  • Entender Quarto como sistema de documentación unificado.
  • Generar múltiples formatos desde un solo source.
  • Preparar material para distintas audiencias.
  • Conectar el experimento de los Días 1–3 con su comunicación.

Mensaje central

Un experimento no existe si no se comunica. El código que nadie lee y los resultados que nadie entiende no tienen impacto. Quarto cierra la brecha entre el pipeline y la audiencia.

2) El problema central

El pipeline de los Días 1–3 produjo:

  • código reproducible ✓
  • métricas documentadas ✓
  • resultados validados ✓

Un resultado necesita llegar a distintas audiencias:

Audiencia Necesita
Revisor académico Paper con metodología detallada
Médico clínico Interpretación práctica
Equipo de producto Deck ejecutivo
Estudiantes Presentación didáctica

El problema sin Quarto

Típicamente: paper en Latex/Word, slides en PowerPoint, código en Jupyter, README en Markdown.

Cuatro herramientas distintas. Cuatro fuentes de verdad. Cualquier cambio en los resultados requiere actualizar cuatro lugares.

3) Quarto como sistema

Un solo source → múltiples salidas.

Cambia: - Diseño y navegación
- Nivel de detalle del código

No cambia: - Contenido y resultados
- Referencias

Comando único:

# Generar todos los formatos
quarto render paper.qmd

# Solo HTML
quarto render paper.qmd --to html

# Solo PDF
quarto render paper.qmd --to pdf

# Preview en vivo
quarto preview paper.qmd

3b) La estructura del proyecto Quarto

quarto/
├── _quarto.yml          ← configuración global
├── index.qmd            ← página de inicio (sitio web)
├── paper.qmd            ← paper principal (HTML + PDF)
├── presentation.qmd     ← slides para alumnos
├── slides_instructor.qmd ← este archivo
├── references.bib        ← bibliografía compartida
└── sections/
    ├── intro.qmd
    ├── methodology.qmd
    ├── results.qmd
    └── discussion.qmd

La clave: {{< include >}}

# En paper.qmd:
`{{< include sections/intro.qmd >}}`
`{{< include sections/methodology.qmd >}}`
`{{< include sections/results.qmd >}}`

Cada sección vive en su propio archivo. paper.qmd las ensambla. Esto permite que distintos colaboradores trabajen en distintas secciones sin conflictos de merge.

Construyendo el documento base

5) Estructura de un paper científico

Las 8 secciones estándar:

  1. Introducción — motivación y pregunta
  2. Pregunta científica — hipótesis falsable
  3. Datos — descripción y preprocesamiento
  4. Metodología — modelos y métricas
  5. Resultados — tablas y figuras
  6. Validación — estabilidad y robustez
  7. Discusión — interpretación y limitaciones
  8. Conclusión — respuesta a la pregunta

En Quarto: cada sección en su propio .qmd.

En nuestro paper:

# paper.qmd
´{{< include sections/intro.qmd >}}´
´{{< include sections/methodology.qmd >}}´
´{{< include sections/results.qmd >}}´
´{{< include sections/discussion.qmd >}}´

# Conclusión {#sec-conclusion}
...

# Referencias
::: {#refs}
:::

Las referencias se insertan automáticamente desde references.bib.

6) Reproducibilidad en documentos

Un documento reproducible integra código y resultados en el mismo archivo.

Celda de código en el paper:

```{python}
#| label: tbl-results
#| tbl-cap: "Métricas en el test set"
#| echo: false

import json
with open("../outputs/metrics/test_metrics.json") as f:
    data = json.load(f)
# ...genera la tabla automáticamente...
```

Cuando se ejecuta quarto render:

  1. Python corre el código
  2. La tabla se genera con los datos reales
  3. Se inserta en el documento

Si los resultados del pipeline cambian, quarto render actualiza el documento automáticamente.

Opciones de ejecución:

execute:
  echo: true      # mostrar código
  eval: true      # ejecutar código
  warning: false  # ocultar warnings
  cache: true     # cachear outputs

Por bloque:

#| echo: false    # no mostrar código
#| eval: false    # no ejecutar
#| fig-cap: "..."

Nota

echo: false + eval: true = figura o tabla sin código visible. Ideal para papers y presentaciones.

7) Diferentes audiencias

El mismo experimento, tres versiones:

Lo que cambia para cada audiencia

Académico Clase General
Ecuaciones Sí Pocas No
Código Opcional Sí (didáctico) No
Referencias Todas Pocas Ninguna
Longitud Largo Medio Corto
Vocabulario Técnico Técnico + explicado Simple

¿Cómo traducir lenguaje técnico a lenguaje general?

Principios clave

  • Habla en ejemplos, no en fórmulas
    → “97 de cada 100 pacientes”

  • Traduce métricas a impacto real
    → ¿qué significa para una persona?

  • Evita símbolos innecesarios
    → reemplaza \(\theta\), recall, AUC

  • Usa comparaciones intuitivas
    → “solo 3 casos se escapan”

Antes vs después

Técnico:

“recall\(_{maligno}\) = 0.976 con θ = 0.40”

General:

“De cada 100 pacientes con cáncer, detecta 97”

Regla práctica

Pregúntate:

¿Esto lo entendería alguien sin background en ML?

Si no → simplifica.

7b) Texto para audiencia general

Versión académica:

“El modelo logró recall\(_{\text{maligno}}\) = 0.976 sobre el conjunto de prueba con umbral de decisión θ = 0.40, superando significativamente al clasificador de mayoría (recall = 0.000).”

Versión general:

“De cada 100 pacientes con cáncer real, el modelo identificó correctamente a 97 de ellas. Solo dejó pasar 3.”

En Quarto, contenido condicional:

::: {.content-visible when-format="html"}
El modelo detecta el 97.6% de los
casos de cáncer. Solo deja pasar
2.4% — aproximadamente 1 de cada 42
mujeres con diagnóstico maligno.
:::

::: {.content-visible when-format="pdf"}
El modelo obtiene recall = 0.976
con AUC-ROC = 0.994, sosteniendo H₁
bajo los criterios establecidos en
la sección de metodología.
:::

Idea clave

Comunicar bien no es simplificar el modelo,
es traducir su impacto al mundo real.

8) Exploración de Quarto — actividad en clase

Los alumnos modifican documentos existentes y observan el resultado.

Ejercicio 1 — cambiar formato:

Ejercicio 2 — ocultar código:

# En paper.qmd, cambiar:

  echo: false   # era: true

¿Qué cambia en el output?

Ejercicio 3 — agregar sección:

Crear sections/appendix.qmd:

# Apéndice: configuración completa

\`\`\`yaml
# configs/baseline.yaml
seed: 42
model:
  type: logistic_regression
  C: 1.0
  class_weight: balanced
\`\`\`

Agregar en paper.qmd:

´{{< include sections/appendix.qmd >}}´

Ejercicio 4 — generar PDF:

quarto render paper.qmd --to pdf

9) Estilos y temas

Temas de HTML (Bootstrap):

format:
  html:
    theme: cosmo    # limpio, moderno
    # theme: flatly  # profesional
    # theme: darkly  # oscuro
    # theme: pulse   # colorido

Ver todos: quarto.org/docs/output-formats/html-themes

Temas de RevealJS:

format:
  revealjs:
    theme: default  # blanco
    # theme: moon   # oscuro
    # theme: sky    # azul
    # theme: serif  # tipográfico

CSS personalizado (conceptual):

format:
  html:
    css: custom.css
  revealjs:
    theme: [default, custom.scss]
/* custom.css */
.callout-tip {
  border-left: 4px solid #1a6633;
}
h1, h2 {
  color: #2166ac;
}

Nota

Hoy usamos temas default. El CSS se añade en la siguiente iteración del documento.

Bonus: publicación en GitHub Pages

10) Publicación en GitHub Pages

# 1. Asegurarse de que _quarto.yml tiene:
#    output-dir: docs

# 2. Renderizar el proyecto completo
quarto render

# 3. Verificar que /docs existe y tiene index.html
ls docs/

# 4. Commit y push
git add docs/
git commit -m "docs: generar sitio Quarto"
git push origin main

# 5. En GitHub: Settings → Pages → Source: main branch, /docs folder

# URL resultante:
# https://usuario.github.io/breast-cancer-classifier/

Flujo recomendado

# Durante el desarrollo:
quarto preview    # servidor local en localhost:4567

# Para publicar:
quarto render && git add docs/ && git commit -m "docs: update"

11) Actividad: construir documento base

Cada alumno construye su propio documento Quarto en 20 minutos.

Paso 1 (5 min) — estructura:

mkdir mi_quarto
cd mi_quarto
quarto create project website

Paso 2 (5 min) — agregar contenido:

Editar index.qmd con: - Título del experimento - Pregunta científica - Un bloque de código Python que cargue los datos

Paso 3 (5 min) — renderizar:

quarto preview

Paso 4 (5 min) — explorar:

Cambiar el tema en _quarto.yml:

format:
  html:
    theme: cosmo

Agregar una sección de resultados con placeholder:

## Resultados

| Métrica | Valor |
|---|---|

| AUC-ROC | 0.994 |

Observar: el mismo .qmd genera HTML, PDF y slides con un solo comando.

12) Cierre

Comunicar es parte del proceso científico.

Un experimento que no se comunica no existe — no puede ser reproducido, criticado, mejorado ni aplicado.
