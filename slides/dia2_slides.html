Recapitulación y pregunta científica

Lo que construimos ayer

El experimento no es el modelo. Es el protocolo completo:

\[\text{Experimento} = \text{H} + \text{D} + \text{M} + \text{P}\]

Hipótesis + Datos + Métrica + Protocolo — todo definido antes de escribir código.

La pregunta científica de hoy

“¿Un modelo de regresión logística con ponderación de clases alcanza recall ≥ 0.90 para la clase maligna y supera a un clasificador de mayoría en AUC-ROC?”

H₀: recall < 0.90 ó AUC-ROC ≤ 0.50

H₁: recall ≥ 0.90 y AUC-ROC > 0.50

Principios de MLOps (1/2): Fundamentos

I. Codebase
Un solo repositorio versionado → múltiples ejecuciones (experimentos, deploys).

II. Dependencies
Dependencias explícitas (requirements.txt, environment.yml) → entornos reproducibles.

III. Config
Configuración fuera del código (YAML/env) → experimentos trazables.

IV. Backing services
Bases de datos, APIs, storage → recursos externos intercambiables.

V. Build, release, run
Separar construir → versionar → ejecutar → evita inconsistencias.

VI. Processes
Procesos sin estado → fáciles de reiniciar, escalar y reproducir.

Intuición

Estos principios aseguran que tu experimento
se pueda reconstruir exactamente en otro entorno.

Principios de MLOps (2/2): Escalabilidad y operación

VII. Port binding
Servicios accesibles vía puertos → despliegue simple y estándar.

VIII. Concurrency
Escalar agregando procesos → paralelización natural en ML.

IX. Disposability
Inicio rápido y apagado limpio → pipelines más robustos.

X. Dev/prod parity
Mismo entorno en desarrollo y producción → menos errores inesperados.

XI. Logs
Logs como flujo de eventos → monitoreo y debugging continuo.

XII. Admin processes
Tareas administrativas como procesos aislados (scripts, evaluaciones).

Idea clave

Un buen sistema de ML no solo aprende,
también se mantiene, escala y se entiende.

¿Por qué organizar un proyecto de forma modular?

Problema (sin modularidad)

  • Código mezclado: datos + modelo + evaluación
  • Difícil de leer y mantener
  • Cambios pequeños → rompen todo
  • Experimentos no reproducibles
  • Hace el proyecto no escalable

Solución (modularidad)

  • Separar responsabilidades claras:
    • datos
    • modelo
    • entrenamiento
    • evaluación
  • Facilita debugging
  • Permite reutilizar componentes

Idea clave

Un buen proyecto de ML no es solo que funcione,
sino que sea entendible, mantenible y reproducible.

Gestión de bibliotecas y entornos

¿Qué significa?

Controlar:

  • Qué librerías usa tu proyecto
  • Qué versiones exactas
  • En qué entorno se ejecuta

Ejemplo:

python=3.11
numpy=1.26.0
scikit-learn=1.4.0

¿Por qué importa?

  • Evita errores tipo:
    • “en mi máquina sí funciona”
  • Garantiza que:
    • Entrenamiento = Evaluación
  • Permite:
    • Reproducir resultados meses después

Idea clave

Sin control de entorno, no hay reproducibilidad real
aunque tengas el mismo código y datos.

¿Por qué usar uv para entornos reproducibles?

Problemas comunes

  • Conflictos entre dependencias

¿Qué aporta uv?

  • Resolución de dependencias determinista
  • Entornos rápidos y ligeros

  • Estructura basada en indentación (como Python)

Ejemplo simple:

¿Por qué este dataset?

sklearn.datasets.load_breast_cancer — Breast Cancer Wisconsin

Característica Valor
Tipo Tabular (30 features numéricas)
Muestras 569
Clases Maligno (212) · Benigno (357)
Desbalance 37% maligno / 63% benigno
GPU No necesaria

El desbalance real (37/63) es lo que justifica class_weight="balanced" y el uso de recall como métrica primaria.

Por qué accuracy engaña aquí

Un modelo que siempre predice “benigno” obtiene:

  • Accuracy = 63% ← parece razonable
  • Recall maligno = 0.0 ← no detecta ningún cáncer

Un cáncer no detectado es mucho más peligroso que una alarma falsa.

→ La métrica que importa es recall de la clase maligna, no accuracy.

División estratificada

# src/data/dataset.py
X_trv, X_test, y_trv, y_test = train_test_split(
    X, y,
    test_size=test_size,
    stratify=y,          # ← misma proporción maligno/benigno en cada split
    random_state=seed,
)

Por qué stratify=y es obligatorio

Sin estratificación, el val set podría tener por azar 20% de malignos en lugar de 37%. Las métricas de validación quedarían sesgadas y la selección del modelo sería injusta.

Blocking por clase = el principio de blocking del Día 1 aplicado a datos tabulares.

Split Muestras % maligno
Train ~384 ~37%
Val ~68 ~37%
Test ~114 ~37%

Bloque 3

Modelo y función de pérdida Regresión logística · class_weight · recall

¿Por qué regresión logística?

Con 569 muestras y 30 features, una red neuronal sobreajustaría fácilmente. La regresión logística es la elección correcta porque:

  • Interpretable: los coeficientes tienen significado directo
  • Rápida: converge en < 100 iteraciones con L-BFGS
  • Probabilística: devuelve P(maligno) ∈ [0, 1]
  • Buen baseline: si no la superas, el modelo complejo no sirve
# src/models/logistic.py
# TODO: entender qué hace class_weight.
# REPLACE: cambia a None y observa
#          el efecto en recall.

LogisticRegression(
    C=1.0,
    class_weight="balanced",
    solver="lbfgs",
    max_iter=200,
    random_state=42,
)

Con class_weight="balanced" sklearn calcula:

\[w_{\text{maligno}} = \frac{n}{2 \times n_{\text{maligno}}}\]

El efecto de class_weight="balanced"

Sin class_weight="balanced"

  • La pérdida trata ambas clases igual
  • El modelo favorece la clase mayoritaria (benigno)
  • Minimiza error global, pero ignora casos críticos

Resultado típico:

  • Accuracy ≈ 92%
  • Recall maligno ≈ 0.85 ✗

Con class_weight="balanced"

  • Los errores en maligno pesan más
  • El modelo prioriza detectar cáncer
  • Sacrifica un poco de accuracy global

Resultado típico:

  • Accuracy ≈ 96%
  • Recall maligno ≈ 0.97 ✓

Idea clave

En problemas desbalanceados, optimizar accuracy puede ser engañoso. El objetivo real es detectar lo importante.

Git y GitHub: ¿qué papel juegan?

¿Qué es Git?

  • Sistema de control de versiones
  • Guarda el historial de cambios en:
    • código
    • configuraciones (YAML)
  • Permite:
    • volver a versiones anteriores
    • comparar cambios
    • trabajar en equipo

¿Qué es GitHub?

  • Plataforma sobre Git en la nube
  • Permite:
    • compartir repositorios
    • colaborar (pull requests)
    • centralizar el proyecto

En ML:

  • Git = memoria del código
  • GitHub = punto de colaboración

Idea clave

Si no versionas tu código,
no puedes saber qué experimento generó qué resultado.

¿Qué es DVC y por qué lo usamos?

DVC (Data Version Control) es una herramienta para:

  • Versionar datos (no solo código)
  • Definir pipelines reproducibles
  • Conectar código + datos + resultados

Extiende a Git → pero para ML

¿Qué problema resuelve?

  • Datos no versionados
  • Resultados que no se pueden reproducir
  • Pipelines manuales y frágiles

Con DVC:

  • Cada paso queda definido
  • Todo se puede re-ejecutar automáticamente
  • Sabes exactamente qué cambió

¿Qué es dvc.yaml?

dvc.yaml es el archivo donde defines:

  • Etapas del pipeline
  • Comandos a ejecutar
  • Entradas y salidas
  • Dependencias (código, datos, configs)

Ejemplo:

stages:
  train:
    cmd: python scripts/train.py
    deps:
      - src/training/trainer.py
      - configs/baseline.yaml
    outs:
      - outputs/model.pkl

¿Qué permite?

  • Ejecutar todo con dvc repro
  • Reproducir solo lo que cambió
  • Trazar:
    • qué script generó qué output
  • Integrar con Git → historial completo

Idea clave

dvc.yaml convierte tu workflow en un
grafo explícito de dependencias reproducible.

Pipeline DVC: tres etapas

# Ejecutar todo de una vez
dvc repro

# Ver qué cambió
dvc params diff
dvc metrics diff

Idea clave

DVC convierte tu experimento en un pipeline reproducible,
no en una serie de scripts sueltos.

Bloque 5

Métricas y evaluación de la hipótesis Recall · AUC-ROC · baseline · H₁

¿Por qué recall como métrica primaria?

\[\text{Recall} = \frac{TP}{TP + FN}\]

Los errores no son simétricos en diagnóstico:

Error Consecuencia
FN (maligno predicho benigno) Cáncer no tratado → peligroso
FP (benigno predicho maligno) Biopsia adicional → molesto, costoso

El costo de un FN es mucho mayor que el de un FP. → Minimizar FN = maximizar recall.

Criterio clínico: recall ≥ 0.90 en maligno.

AUC-ROC como métrica secundaria

AUC-ROC mide la capacidad discriminativa independientemente del umbral.

  • AUC = 0.50 → no discrimina (igual que una moneda)
  • AUC = 1.00 → separación perfecta

Un modelo con recall = 1.0 y AUC = 0.5 predice siempre maligno. Por eso exigimos AUC > 0.50 como condición adicional.

El baseline de mayoría

# src/evaluation/metrics.py

dummy = DummyClassifier(strategy="most_frequent")
dummy.fit(X_train, y_train)

# Predice siempre "benigno" (clase mayoritaria)
# accuracy  = 0.626  ← parece razonable
# recall    = 0.000  ← no detecta ningún cáncer
# AUC-ROC   = 0.500  ← no discrimina en absoluto

El baseline define el piso mínimo

Cualquier modelo útil debe superar al baseline en ambas métricas simultáneamente.

Si accuracy = 96% pero recall = 0.80: el modelo no sirve para screening clínico. Si recall = 0.95 pero AUC = 0.50: el modelo solo predice siempre positivo.

La hipótesis H₁ exige que se cumplan las dos condiciones a la vez.

Evaluación de la hipótesis

# src/evaluation/metrics.py
def print_report(metrics, baseline_metrics, cfg):
    target = cfg["metrics"]["target"]   # 0.90
    recall = metrics["recall_malignant"]
    auc    = metrics["auc_roc"]

    h1_recall = recall >= target        # recall ≥ 0.90
    h1_auc    = auc    > 0.50
    h1        = h1_recall and h1_auc

    print(f"Recall maligno : {recall:.4f}  → {'✓' if h1_recall else '✗'}")
    print(f"AUC-ROC        : {auc:.4f}  → {'✓' if h1_auc else '✗'}")

    if h1:
        print("→ H₁ SOSTENIDA")
    else:
        print("→ H₀ NO RECHAZADA — ajustar class_weight o C")
